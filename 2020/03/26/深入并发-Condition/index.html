Condition概念

JUC 包提供了 Condition 来对锁进行精准控制，Condition 是一个多线程协调通信的工具类，可以让某些线程一起等待某个条件(condition)，只有满足条件时，线程才会被唤醒。Condition的本质就是等待队列和同步队列的交互。

源码分析

如果不熟悉 AQS 的同学建议先去看楼主的上一篇博文《深入并发-AQS》，不然理解本章将很吃力。
下面是对 Condition 的一个基本使用

await 方法分析

我们从 condition.await() 开始源码分析

  1. 创建一个新的节点，节点状态为 condition，采用的数据结构仍然是链表并将其添加至等待队列的尾结点
  2. 释放当前的锁，得到锁的状态，并唤醒 AQS 队列中的一个线程
  3. 如果当前节点没有在同步队列上，即还没有被 signal，则将当前线程阻塞
  4. 其他线程调用 condition.signal() 唤醒当前线程或当前线程被执行 interrupt 方法进行中断从而唤醒
  5. 判断线程是如何被唤醒的，如果该线程在阻塞中被执行中断从而导致醒来需要抛出异常 interruptMode = THROW_IE，如果是其他线程唤醒则再次调用 thread.interrupt() 方法进行中断 interruptMode = REINTERRUPT
  6. 线程尝试获取锁, 当 acquireQueued 返回 false 就是拿到锁了
  7. interruptMode != THROW_IE 表示这个线程没有成功将 node 入队,但其他线程在执行 signal 方法时会执行 enq 方法让其入同步队列
  8. 如果 node 的下一个等待者不是 null, 则进行清理,清理 Condition 队列上的节点
  9. 根据 interruptMode 的值进行中断或者抛出异常

我们来看下插入到等待队列的尾结点方法 addConditionWaiter，注意等待队列是一个单向链表（Node节点的 next 和 prev 是同步队列节点的属性）

  1. 获取当前尾节点，如果尾结点为空或状态异常则清除等待队列中被取消的节点并重新获取尾结点
  2. 将当前线程封装成一个 Node 节点，将之前尾结点的 nextWaiter 设置为当前节点
  3. 如果尾结点为空，则 node 节点即是尾结点也是头节点

接着我们来看下释放当前锁的方法 fullyRelease

  1. 首先获取到锁的重入次数，
  2. 将重入次数通过 CAS 更新为0并将锁的当前所有者置为 null
  3. 唤醒同步队列中的下一个节点
  4. 返回锁的重入次数用于后续唤醒时重新设置 state 的值
  5. 最后如果释放锁失败则将当前线程节点的状态修改为取消

看下如何判断线程对应的节点是否在同步队列中

  1. 判断节点的值是否为 CONDITION (同步队列不存在 CONDITION 状态)或 node.prev 为空(在 AQS 队列中只有 head 节点的 prev 会为 null，head 节点意味着它是获得锁的节点)，如果是则直接返回 false
  2. 如果后置节点不为空(等待队列是一个单向链表，通过 nextWaiter 获取下一个节点)，则直接返回 true 。
  3. 通过遍历同步节点判断是否包含当前节点

acquireQueued 方法在 AQS 那篇文章已经分析过，主要就是通过自旋的方式抢占锁。

signal 方法分析

从 condition.signal() 进入分析

  1. 判断当前线程是否锁的持有者，不是则抛出异常
  2. 获取等待队列的首节点进行唤醒

我们来看下方法 doSignal

  1. 如果等待队列首节点的下一个节点是 null, 那么最后一个节点也是 null
  2. 对等待队列中从首部开始的第一个 condition 状态的节点，执行 transferForSignal 操作，将 node 从等待队列中转换到同步队列中，同时修改同步队列中原先尾节点的状态

transferForSignal 方法是唤醒的具体代码实现

  1. 将当前节点的 waitStatus 通过 CAS 设置为0，如果设置失败说明该节点已经被CANCELLED，直接返回 false
  2. 如果设置成功，则将当前节点添加到同步队列的尾结点
  3. 如果上一个节点的状态为 CANCELLED 或者尝试设置上一个节点的状态为 SIGNAL 失败(SIGNAL 表示: 他的 next 节点需要停止阻塞)则直接唤醒输入节点上的线程。
  4. 如果 node 的 prev 节点已经是 signal 状态，那么被阻塞的线程的唤醒工作由 AQS 队列来完成

总结

线程 awaitThread 先通过 lock.lock()方法获取锁成功后调用了 condition.await 方法进入等待队列，而另一个线程 signalThread 通过lock.lock()方法获取锁成功后调用了 condition.signal 或者 signalAll 方法，使得线程 awaitThread 能够有机会移入到同步队列中，当其他线程释放 lock 后使得线程 awaitThread 能够有机会获取 lock，从而使得线程 awaitThread 能够从 await 方法中退出执行后续操作。如果 awaitThread 获取 lock 失败会直接进入到同步队列继续等待获取锁。

阻塞：await()方法中，在线程释放锁资源之后将当前线程节点添加到等待队列的尾结点并进入阻塞状态。等待其他线程唤醒添加到同步队列中，最后通过自旋的方式尝试获取锁。
释放：signal()方法中，将线程节点从 condition 等待队列移动到 AQS 同步队列，并唤醒当前节点线程进入正常锁的获取流程。
